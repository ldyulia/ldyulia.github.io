오브젝트에 프로퍼티 추가

ES5부터는 프로퍼티를 무작정 변경할 수 없도록 제약이 생겼다. 각 프로퍼티마다 상태를 갖고 있으며 변경을 허용하는 상태이면 값을 변경할 수 있지만 변경 불가 상태면 값을 변경할 수 없다. 프로퍼티 추가시 다음 변경을 위해 권한을 설정해야 한다. 권한을 설정하지 않으면 삭제, 변경을 할 수 없으며 for-in문으로 읽을 수도 없다.

defineProperty() 함수 : 오브젝트에 프로퍼티를 추가하거나 프로퍼티 속성을 변경하는 함수

첫 번째 파라미터에 추가, 변경할 대상 오브젝트를 작성

두 번째 파라미터에 추가, 변경 대상 프로퍼티 이름을 작성

세 번째 파라미터에 속성 이름과 속성 값을 작성

value는 약속된 이름으로 다른 이름을 사용할 수 없으며 속성이라고 부른다.

defineProperty함수가 실행되면 obj오브젝트에 book프로퍼티가 없으므로 새로운 프로퍼티로 추가되게 되며 {‘book’ : 123}형태가 된다. book프로퍼티가 기존에 존재했다면 값이 123으로 변경되게 된다. 이 때, book프로퍼티가 변경 허용 상태일 때만 변경할 수 있으며 변경 불가 상태이면 변경할 수 없다.

프로퍼티 디스크립터 타입 (Property Descriptor Type)

프로퍼티의 속성을 구체적으로 정의하거나 처리 기준을 정의한다. 속성 기술자, 속성 서술자 라고도 쓰임.

프로퍼티 디스크립터 타입에는 데이터(data) 프로퍼티 디스크립터와 액세스(access) 프로퍼티 디스크립터가 있다.

프로퍼티 디스크립터는 아래와 같이 필드이름과 필드값으로 구성되면 이를 레코드(record)라고 한다.

즉, 필드이름 enumerable의 필드 값이 true일 때, 레코드는 “enumerable : true”가 되는 것이다.

데이터 프로퍼티 디스크립터
필드이름 필드값 (디폴트 값)
value undefined
writable false
enumerable false
configurable false
액세스 프로퍼티 디스크립터
필드이름 필드값 (디폴트 값)
get undefined
set undefined
enumerable false
configurable false

각 프로퍼티 디스크립터에 속한 필드만 같이 사용할 수 있다. 예를 들어 데이터 프로퍼티 디스크립터인 value와 액세스 프로퍼티 디스크립터인 get또는 set은 함께 쓰일 수 없다. enumerable과 configurable은 공통으로 사용할 수 있다.

프로퍼티 디스크립터 인식 기준

자바스크립터는 먼저 value와 writable작성을 체크한다. value 또는 writable이 작성되어 있으면 데이터 프로퍼티 디스크립터로 인식하고, 작성되어 있지 않으면 액세스 프로퍼티 디스크립터로 인식한다. value와 get을 함께 작성하면 데이터 프로퍼티 디스크립터로 인식하는 것이 아니라 에러가 발생한다.

value

오브젝트의 프로퍼티 값으로 사용될 값. {value : ‘abc’}형태로 작성한다. 작성된 필드 값은 for-in문으로 읽을 때 프로퍼티 값이 된다.

writable

writable: false이면 value의 필드 값 변경 불가, true이면 가능

enumerable

enumerable : false이면 for-in문으로 열거 불가, true이면 열거 가능

결과 undefined

enumerble을 설정해주지 않으면 디폴트값인 false로 설정되기 때문에 for-in문을 통해 열거할 수 없다.

결과 book:123

configurable

configurable : false이면 오브젝트의 프로퍼티를 삭제할 수 없으며 true이면 삭제 가능. 또한 false이면 value이외의 속성을 변경할 수 없으며 true이면 변경 가능

get속성

get 속성은 속성 값에 함수 또는 undefined를 작성할 수 있다.

위 코드를 콘솔창에서 작성해보면 obj오브젝트에 book프로퍼티와 get함수가 설정이 된 것을 확인 할 수 있다. 이때 get book : f() 형태로 표시된 것은 get함수가 book프로퍼티에 연결되었다는 것을 의미한다.

자바스크립트 엔진이 obj.book; 문장을 만나면 book프로퍼티의 값을 반환하는 것이 아니라 obj.book프로퍼티의 get함수를 호출한다. 이것이 get함수의 특징이다.

obj.book.get();형태로 호출하면 에러가 발생한다. obj.book.get();의 형태로 함수를 작성하려면 코드를 아래와 같이 작성해야 한다. defineProperty함수로 get속성에 함수를 작성하는 것은 다른 개념이며 구조이다.

set 속성

set 속성 또한 속성 값에 함수 또는 undefined를 작성할 수 있다.

코드 해석
  1. get함수가 호출되면 bookValue 라는 변수값을 반환한다.
  2. set함수가 호출되면 파라미터로 받은 값을 bookVale 변수에 할당한다.
  3. 자바스크립트 엔진이 obj.book = 12345; 문장을 만나면 obj.book프로퍼티에 12345를 할당하지 않고 set함수를 호출한다. 이 때 12345를 파라미터 값으로 넘겨주는 것이다. 호출된 set함수에서 파라미터 값인 12345를 bookValue변수에 할당한다.
  4. 자바스크립트 엔진이 obj.book;문장을 만나면 get함수를 호출해 set함수에서 bookValue변수에 설정한 12345를 반환한다. 이런 과정을 통해 실행결과 12345가 출력되는 것이다.

빌트인(built-in)

전형적인 객체지향 프로그램에서 특정 오브젝트에 속한 메서드를 호출하기 위해서는 아래와 같이 먼저 오브젝트 생성을 해주어야 한다.

그러나 자바스크립트는 렌더링 단계에서 해당 오브젝트들(String, Array 등)을 먼저 생성해두기 때문에 위와 같이 해당 오브젝트를 일일이 생성할 필요가 없다. 이와 같은 오브젝트를 빌트인 오브젝트라고 한다.

이처럼 사용자가 일일이 세팅해주지 않아도 사용할 수 있도록 자바스크립트 렌더링 단계에서 미리 만들어지는 모든 것을 총칭하는 것이 바로 빌트인이다. 이러한 빌트인에는 크게 연산자, 데이터타입, 오브젝트가 있다.

자바스크립트 용어정리 1

컴파일러(Compiler)

어떤 프로그래밍 언어로 쓰여진 파일을 다른 언어로 바꿔주는 번역 프로그램. 대체로 고수준 언어를 기계어로 번역하는 프로그램을 의미. 프로그램을 작동시키기 위해서는 컴퓨터가 알아듣는 기계어로 작성을 해야하는데, 처음부터 기계어로 프로그래밍을 하기에는 너무 어렵고 비효율적. 따라서 인간에게 편리한 고수준의 추상화된 언어로 프로그래밍을 하면 이를 기계가 알아들을 수 있는 기계어로 번역하기 시작했고, 이 역할을 하는 도구가 바로 컴파일러다.

스크립트 언어

프로그래밍 언어의 한 종류. (컴파일 언어처럼) 컴파일 과정을 거치지 않고 소스코드를 한 줄 한 줄 바로바로 읽어 내려가는 인터프리터 방식의 언어. 프로그래밍 언어라고는 하지만 엄밀한 의미로 응용프로그램을 만드는 것 보다는, 기존에 존재하는 프로그램 위에서 구동되며 그 기존의 프로그램을 제어하기 위해 쓰인다. 웹 브라우저에서 구동되며 웹브라우저를 제어하기 위해 사용되는 자바스크립트가 대표적.

인스턴스(instance)

쉽게 말해 원본(프로토타입 객체)의 복사본이다. 복사본이기 때문에 원본의 속성(프로퍼티와 메소드)을 공유(상속)하면서도 개별적으로 추가된 다른 속성들을 가질 수 있다.

프로토타입(Prototype)

인스턴스(복사본)을 찍어내기 위해 만드는 원본 틀이라고 생각하면 쉽다. 자바스크립트의 클래스(Class)는 같은 프로토타입 객체로부터 프로퍼티를 상속받은 객체의 집합이다. 따라서 프로토타입 객체는 클래스의 핵심이라고 할 수 있다.

생성자(constructor)함수

생성자는 새로 생성된 객체를 초기화하는 용도로 사용되는 함수다. 생성자는 new키워드를 사용하여 호출된다. 즉, new키워드 뒤에는 반드시 함수가 호출되어야 하는데, 이때 호출되는 함수를 생성자 함수라 하는 것이다. 아래의 코드는 자바스크립트에 기본적으로 내장된 생성자 함수이다.

위와같이 미리 내장된 생성자 뿐만 아니라 사용자가 원하는 형태의 객체를 생성하고 초기화하기 위해 임의의 생성자함수를 직접 정의할 수도 있다.

클래스와 생성자 함수의 이름은 보통 대문자로 시작하는데(일반 함수와 메서드는 소문자로 시작), 이는 생성자 호출이 일반 함수 호출과 크게 다르기 때문이다. 생성자는 보통 new키워드를 사용하여 호출된다고 가정하기 때문에, 일반적인 함수 호출처럼 호출하면 보통 제대로 작동되지 않는다. 따라서 일반적인 함수와 구분되도록 작명 규칙을 지켜야 언제 new키워드를 이용해야 하는지 알 수 있다.

함수와 메서드

객체란 서로 연관된 변수와 함수를 그룹핑한 것이라고 볼 수 있다. 즉, 객체 내의 변수를 프로퍼티 네임(속성 또는 속성 이름)이라 하고, 이 프로퍼티 네임이 가진 속성 값(value)이 함수일 때, 이 함수를 메서드라 부르는 것이다.

쉽게 말해 메서드는 객체가 가지고 있는 동작이라고 보면 된다. 함수와 구분하여 부르는 이유는, 메서드는 반드시 *객체를 통해서만 *동작을 수행할 수 있기 때문이다. 즉, 동작 수행의 주체가 객체인 셈이다.

전역변수와 지역변수 그리고 객체지향프로그래밍까지

프로그래밍을 할 때, 같은이름의 변수 사용으로 코드가 꼬이는 경우가 빈발한다. 이를 해결하기 위해 나온 개념이 바로 전역변수와 지역변수 개념. 쉽게 말해 폴더정리를 하는 것과 같은 것이다. 같은 폴더 안에서는 같은 이름을 가진 파일이 존재할 수 없지만(덮어쓰기 되어버림), 서로 다른 폴더 안에서는 같은 이름을 가진 파일이 존재해도 상관없다. 그리고 이런 흐름으로, 문제 해결을 위해 발전해온 것이 바로 ‘객체지향 프로그래밍’이다.

스코프(scope)

scope = 유효범위 : 변수의 수명

전역변수(global) : 코드 전역에서 접근 가능한 변수

지역변수(local) : 함수 안에서 생성되고 함수 안에서만 접근 가능한 변수 (자바스크립트의 지역변수는 함수 안에서만 생성 가능하다. for문이나 if문 등에서 생성된 변수는 지역변수로 기능하지 못함)

cf) ES5에서 사용되던 var는 스코프가 함수단위이지만, ES6부터 도입된 const와 let은 스코프가 블록단위이다.

반드시 전역변수를 써야 하는 특별한 이유가 없는 한 지역변수를 써야 한다.

그러나 불가피하게 전역변수를 써야 할 경우 다음과 같이 객체를 이용해 전역변수를 만들고 관리한다.

위와 같이 코드를 작성하면 MYAPP이라는 전역변수 하나만을 생성하여 그 안에 필요한 다른 변수(=속성 : 객체 안에 들어있는 변수를 속성이라 부른다)를 지정하는 식으로 관리할 수 있다. 즉 MYAPP이라는 단 하나의 전역변수 안에 자신이 필요한 다른 전역변수들을 넣어놓음으로써 코드 충돌을 방지할 수 있는 것이다.

만약 위와 같이 MYAPP이라는 전역변수 하나 조차 사용하고싶지 않다면 아래와 같이 익명함수를 이용하여 작성하면 된다.

위와 같은 방식은 제이쿼리를 비롯한 수많은 라이브러리에서 모듈화를 시킬 때 자주 사용하는 기법이다.

API에 대해 공부하다 알게 된 새삼스러운 사실

API : Application Programing Interface

UI(User Interface)가 일반 사용자가 시스템을 제어할 수 있게 해주는 인터페이스(버튼, 경고박스 등)라면, API는 개발자들이 시스템을 제어할 수 있게 해주는 인터페이스(경고창을 웹브라우저에 띄울 수 있게 해주는 명령어 alert 등)

라는 것 까지는 놀라울 거 하나 없는 사전적 지식이었는데 그래서 대체 Application과 Interface는 무엇인가에 대해서는 한 번도 생각해본 적 없어서 깨닫고 난 뒤 매우.. 새삼스러웠다.

일단 일상에서 별 생각없이 응용프로그램 응용프로그램 어플리케이션 어플리케이션 소리를 잘도 나불대고 다녔으면서도 실은 응용프로그램 = Application의 한국식 이름 이라는 것까지는 생각해본 적 없었고..

그래서 대체 뭐가 뭐를 응용한다는 건지..? 까지는 당연히 생각해봤을 리가 없다

그래서 대체 뭐가 뭐를 응용한단 말인가?

근까 쉽게 말해 이런 거였다

  1. 사용자가 웹브라우저를 이용하기 위해서는
  2. 웹개발자들이 웹을 구현해줘야 하고
  3. 웹개발자들이 웹을 구현하기 위해서는 브라우저 개발자들이 브라우저를 만들어줘야 하고
  4. 브라우저 개발자들이 브라우저를 만들기 위해서는 운영체제 개발자들이 운영체제를 만들어줘야하고
  5. … 아무튼 이런 식으로 쭉쭉 아래까지 내려가다보면 전자공학도 만나고 전기공학까지 만나게 된다

프로그래밍은 사실 위와 같이 계층적인 구조로 이루어져 있는데 각각의 영역이 맞닿는 지점을 바로 Interface라고 한다는 거다..! 그리고 웹개발자는 브라우저를 ‘응용’해서 프로그래밍을 하고, 브라우저 개발자는 운영체제를 ‘응용’해서 프로그래밍을 하기 때문에 ‘응용프로그램’이라고 한다는 거였다!

즉, 어떤 기반프로그램의 인터페이스를 이용해 응용프로그램을 만들고, 이 응용프로그램이 또 다른 응용프로그램의 기반프로그램이 되는 식으로 계층적으로 연결이 되는 것이다. WOW

정말 새삼스러운 사실이었음을..
